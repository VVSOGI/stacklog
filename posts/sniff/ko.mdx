# 이 프로젝트는 무엇인가?

형제의 팀 프로젝트입니다. 실제 저희 형과 같이 진행한 프로젝트입니다. 어떤 아이디어로 프로젝트를 해볼까 하다가 반려동물 그 중에서도 반려견이 현대에서 정서적으로 중요한 역할을 하고 있고, 사람들이 반려견을 위한 서비스로 뭘 찾고 있을까 하다가 나온 아이디어로 반려견 산책 대행 서비스를 만들게 됐습니다. 기획과 초기 API 설계는 공동으로 작업했고, 피그마를 이용한 디자인 작업 및 프론트엔드 리드를 맡았습니다.

## 기능은?

위치 기반 서비스 개발을 위한 KAKAO Maps API, 디바이스에 따른 반응형 디자인 최적화, 회원 가입 및 로그인에 구글 SSO를 도입해 UX를 개선했습니다. Lighthouse의 웹페이지 품질 측정에서 초록색 양호 점수를 받았습니다. 서비스가 한국인으로 국한되지 않게 하기 위해 미들웨어를 통해 다국어 지원 시스템을 구축했습니다.

프론트에서 백엔드로 API를 요청할 때 Next js 자체 서버를 BFF 서버 형식으로 사용했습니다. 이를 통해서 백엔드에서는 순수하게 비즈니스 로직 관련 작업을 할 수 있고, CORS 에러 또한 피할 수 있습니다. 또한 서드 파티 API 또한 Next 서버에서 접근할 수 있기에 Next 서버를 적극적으로 활용했습니다.

또한 배포 프로세스를 줄이기 위해서 도커와 연계해 명령어 한 번이면 배포가 가능하게끔 배포 자동화 shell scripts를 작성했습니다. 이는 추후에 Github Actions CI CD에 연동해서 사용하게 됩니다.

## 아키텍처에 대한 설명

AWS EC2에 도커 컨테이너로 Nginx, 프론트, 백엔드, 데이터베이스가 관리되고 있습니다. 각 서비스들에 대한 요청은 Nginx의 리버스 프록시를 이용해 내부 로컬 서비스들로 연결되고 서비스들은 도커 네트워크를 통해서 내부적으로 소통을 하고 있습니다. 각 서비스에서 발생하는 로그는 Nginx를 통해서 계속해서 쌓이게 되는데 Ubuntu의 logrotate를 사용하여 일자별로 로그의 용량을 확인 후 설정한 용량보다 높다면 해당 로그에 날짜를 작성한 후 다른 이름으로 저장합니다. 이 파일이 14개를 초과하면 가장 오래된 데이터 순서대로 하나씩 제거합니다.

도메인은 Route 53을 통해서 구매했고, SSL 인증서는 가비아를 통해서 구매했습니다. SSL, 도메인 연결은 Nginx를 이용해서 적용했습니다.

정적 데이터는 S3를 통해서 관리되고 있으며 S3는 현재 그림에 나와있지 않지만 데이터 호출 시 클라우드 프론트를 통해서 이미지를 제공하고 있습니다. 이를 통해 버킷의 구조 노출을 가렸고, 컨텐츠를 유저에게 데이터를 더 빠르게 제공할 수 있게 됐습니다.

## 소통은 어떻게?

간단한 일정과 정보 공유는 슬랙을 사용했고, 더 자세한 일정은 노션을 사용했습니다. 스프린트 칸반 보드를 만들어서 Backlog, In progress, Complete로 나눠서 일정 관리를 했습니다. 프로젝트 진행 중 서로에게 도움이 될만한 이슈는 Issue Share라는 표를 만들어서 발생한 문제, 발생 시간, 해결 방법을 상세하게 작성했습니다. 칸반 보드 60개가 넘어갈 때 쯤 프로젝트를 마무리했습니다.
